# Multi DB React Agent Specification

## Problem Statement

SkyNova Airlines needs an internal AI assistant that can answer operational and policy questions from multiple data sources without asking the user to know where the answer lives.

The agent should accept a natural language question, decide which backend tool or tools are relevant, call those tools, and return a clear business-friendly answer. It should also expose the tool call trace so a demo user can see which data source was used and what came back.

## Users and Question Types

The primary user is an airline operations, support, or business analyst user. They are not expected to know SQL, MongoDB query syntax, vector search, or the internal data model.

Typical questions include:

- Flight operations questions, such as cancelled flights, delayed flights, routes, aircraft, fares, and passenger bookings.
- Customer support questions, such as high-priority tickets, ticket status, customer issues, and support activity.
- Review and activity questions, such as low-rated flight reviews, customer feedback, and user activity logs.
- Policy questions, such as baggage rules, refund procedures, check-in policy, compensation, and frequent flyer rules.
- Mixed-source questions that require structured records and policy context, such as delayed flights plus compensation rules.

## Tool Contracts

### Tool 1: `sql_query`

Purpose:

Use this tool for structured PostgreSQL questions about flights, passengers, and bookings.

Input arguments:

```text
question: str
```

The input is the user's natural language question or a focused sub-question generated by the agent.

Data scope:

- `passengers`
- `flights`
- `bookings`

Return shape:

The tool returns a JSON string with this shape:

```json
{
  "sql_executed": "SELECT ... LIMIT 20",
  "row_count": 1,
  "data": [
    {
      "flight_number": "SN404",
      "origin": "MIA",
      "destination": "JFK",
      "departure_time": "2025-06-03 15:00:00+00:00",
      "arrival_time": "2025-06-03 18:00:00+00:00",
      "status": "cancelled",
      "aircraft_type": "Airbus A321",
      "base_price": "210.00"
    }
  ],
  "warnings": [
    "No LIMIT clause found - results may be large."
  ]
}
```

Notes:

- `warnings` is optional and appears only when warnings are generated.
- The tool auto-injects `LIMIT 20` when a generated SELECT query does not include a limit, except for count-style queries.
- The PostgreSQL query is executed with a statement timeout.
- The tool returns at most 20 rows in `data`.

Error surfacing:

- Non-SELECT SQL raises an error.
- Dangerous SQL keywords such as `INSERT`, `UPDATE`, `DELETE`, `DROP`, `CREATE`, `ALTER`, `TRUNCATE`, `GRANT`, `REVOKE`, `EXECUTE`, and `EXEC` raise an error.
- Database connection or execution failures surface as tool/runtime errors to the agent and then to the API error path.

### Tool 2: `mongo_query`

Purpose:

Use this tool for MongoDB questions about support tickets, flight reviews, and user activity logs.

Input arguments:

```text
question: str
```

The input is the user's natural language question or a focused sub-question generated by the agent.

Data scope:

- `support_tickets`
- `flight_reviews`
- `user_activity_logs`

Return shape:

The tool returns a JSON string with this shape:

```json
{
  "collection": "support_tickets",
  "filter_used": {
    "priority": "high"
  },
  "count": 2,
  "data": [
    {
      "ticket_id": "TKT-001",
      "customer_id": "SN-10001",
      "customer_name": "Example Customer",
      "flight_number": "SN101",
      "subject": "Missing baggage",
      "description": "Customer reported missing baggage after arrival.",
      "status": "open",
      "priority": "high",
      "created_at": "2025-06-01T10:00:00",
      "tags": ["baggage"]
    }
  ]
}
```

Safety behavior:

- The selected collection must be in the whitelist.
- The filter must be a JSON object.
- Results are capped with `.limit(20)`.
- Server-side JavaScript operators are blocked, including `$where`, `$function`, `$accumulator`, and `$expr`.
- Unrecognized MongoDB operators are rejected.

Error surfacing:

Mongo safety and validation errors are returned as JSON strings with an `error` field:

```json
{
  "error": "Collection 'payments' is not allowed."
}
```

or:

```json
{
  "error": "Dangerous MongoDB operator '$where' is not allowed."
}
```

Database connection or execution failures surface as tool/runtime errors to the agent and then to the API error path.

### Tool 3: `handbook_search`

Purpose:

Use this tool for SkyNova policy, rule, procedure, refund, baggage, boarding, check-in, compensation, and frequent flyer questions.

Input arguments:

```text
question: str
```

The input is the user's natural language question or a focused policy search query generated by the agent.

Data scope:

- `handbook_chunks` in PostgreSQL with pgvector embeddings.
- Embedding model: `text-embedding-3-small`.
- Top K chunks: 5.

Return shape:

The tool returns a JSON string with this shape:

```json
{
  "source": "SkyNova Passenger Handbook",
  "top_chunks": [
    {
      "source": "skynova_handbook.txt",
      "content": "Passengers may be eligible for compensation when...",
      "similarity": 0.82
    }
  ]
}
```

Error surfacing:

- Embedding generation failures surface as tool/runtime errors.
- PostgreSQL or pgvector query failures surface as tool/runtime errors.
- Empty retrieval results should be summarized by the agent as no matching handbook context found.

## Routing Rules

The system prompt tells the agent that it is a helpful assistant for SkyNova Airlines with three tools:

- `sql_query`: use for questions about flights, passengers, and bookings in PostgreSQL.
- `mongo_query`: use for questions about support tickets, flight reviews, and user activity logs in MongoDB.
- `handbook_search`: use for questions about airline policies, rules, refunds, baggage, check-in, frequent flyer program, and procedural information.

The prompt also instructs:

- Always use the most specific tool.
- If a question spans multiple sources, call each relevant tool in turn.
- Write a user-friendly final answer for a business user.
- Start with the direct answer, then use concise bullets or short labeled lines when useful.
- Do not use Markdown tables.
- Do not expose raw JSON, database rows, internal tool names, or implementation details unless the user asks for them.
- If tool output is empty, say what was checked and that no matching records were found.

In addition to the prompt, the backend applies deterministic keyword routing before creating the agent run:

- SQL keywords include flight, booking, passenger, aircraft, departure, arrival, origin, destination, seat, cabin, tier, delayed flight, cancelled flight, scheduled flight, confirmed, and base price.
- MongoDB keywords include support ticket, ticket, priority, review, flight review, low rating, food rating, activity log, user activity, device, web, and mobile.
- Handbook keywords include policy, refund, baggage, check-in policy, boarding, compensation, fee, frequent flyer program, rule, procedure, and handbook.

If no keyword route matches, the agent receives all three tools and decides which one to use.

## Definition of Done

The system is done when:

- The user can sign in and submit a natural language question.
- The backend protects chat endpoints with JWT auth.
- The agent selects the correct tool or tools for the question.
- The response includes a user-friendly final answer.
- The UI displays the user question, final answer, and tool call trace.
- SQL and MongoDB safety checks are active.
- The golden evaluation runner can test representative questions and record results.

## Sample Questions and Expected Answer Shape

### 1. Recently cancelled flights

Question:

```text
Show me recently cancelled flights.
```

Expected tool call:

```text
sql_query
```

Expected answer shape:

- Direct sentence summarizing cancelled flight records found.
- Flight number.
- Origin and destination.
- Departure and arrival time.
- Status.
- Aircraft type.
- Base price if present.
- No raw JSON and no Markdown table.

### 2. High-priority support tickets

Question:

```text
Which support tickets are high priority?
```

Expected tool call:

```text
mongo_query
```

Expected answer shape:

- Direct sentence summarizing high-priority ticket count or matches.
- Ticket ID.
- Customer name or customer ID.
- Flight number if present.
- Subject or issue summary.
- Ticket status.
- Priority.

### 3. Refund policy

Question:

```text
What is the refund policy for cancelled flights?
```

Expected tool call:

```text
handbook_search
```

Expected answer shape:

- Direct policy summary.
- Eligibility or condition details.
- Important timing or procedural notes.
- Any fees or exceptions if retrieved.
- Should be grounded in handbook content.

### 4. Low-rated flight reviews

Question:

```text
Find low-rated flight reviews and summarize the issue.
```

Expected tool call:

```text
mongo_query
```

Expected answer shape:

- Direct summary of low-rated reviews found.
- Review ID or flight number.
- Route or travel date if present.
- Overall rating.
- Main complaint from review text or ratings.
- Recommendation flag if present.

### 5. Delayed flights and compensation policy

Question:

```text
Show delayed flights and tell me the compensation policy.
```

Expected tool calls:

```text
sql_query
handbook_search
```

Expected answer shape:

- First section summarizing delayed flight records.
- Flight number.
- Route.
- Departure and arrival time.
- Status.
- Second section summarizing compensation policy.
- Eligibility or procedure from handbook context.
- Clear separation between operational flight facts and policy guidance.
